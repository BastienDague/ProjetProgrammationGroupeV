\section{Introduction}
\subsection{Contribution Open Source DTN-MTP}
Le projet s’inscrit dans un contexte open source, ce qui a fortement influencé notre manière de travailler. Contrairement à un projet strictement scolaire, les contributions ne sont pas simplement ajoutées à un dépôt privé : elles doivent s’intégrer proprement dans un écosystème existant, être relues, discutées, corrigées puis validées par les mainteneurs du projet. Dans notre cas, les modifications apportées à A-SABR, DT-Chat ou BP-Socket ont donc été réalisées sous forme de branches puis de Pull Requests, avec des échanges réguliers autour de la qualité du code, de la maintenabilité et de la cohérence avec l’existant.

A-SABR, DTChat et BP-Socket font partie de l’écosystème DTN-MTP, qui regroupe plusieurs outils liés aux réseaux tolérants aux délais. A-SABR constitue la bibliothèque de routage, DT-Chat sert d’interface graphique permettant d’expérimenter et de visualiser des communications DTN, tandis que BP-Socket vise à fournir une interface socket adaptée au Bundle Protocol. Ces différents dépôts ne sont donc pas indépendants : ils participent à un ensemble logiciel plus large, où les évolutions d’un composant peuvent avoir des conséquences sur les autres.

Ce contexte open source nous a confrontés à des problématiques concrètes de développement collaboratif. Il ne suffisait pas d’écrire du code fonctionnel : il fallait aussi respecter les conventions du projet, maintenir la compatibilité avec les autres dépôts, documenter les changements, répondre aux retours et parfois adapter notre travail à des refactorisations récentes. Cette organisation nous a permis de mieux comprendre le fonctionnement d’un projet logiciel réel, dans lequel la qualité d’intégration est aussi importante que la fonctionnalité développée.
\subsubsection{DTN}
Les réseaux \gls{dtn} constituent encore une approche relativement récente par rapport aux réseaux classiques, mais ils sont déjà utilisés dans des contextes réels, notamment pour le rapatriement de certaines données scientifiques depuis l'ISS. Leur intérêt est particulièrement fort dans le domaine spatial, où une connexion continue entre les nœuds ne peut pas toujours être garantie. Contrairement aux réseaux traditionnels, qui supposent souvent l'existence d'un chemin de bout en bout entre l'émetteur et le récepteur, les réseaux DTN acceptent les délais importants, les interruptions de connexion et les périodes d'indisponibilité. Les réseaux spatiaux du futur reposeront de plus en plus sur ce type de technologie.

Dans ce type de réseau, les données sont transmises selon un principe de stockage puis retransmission : un nœud peut conserver temporairement un message, appelé \gls{bundle}, puis le transmettre plus tard lorsqu’un contact avec un autre nœud devient disponible. Ce fonctionnement est particulièrement adapté aux communications spatiales, où les distances sont importantes, les délais de propagation élevés et les fenêtres de communication limitées.

Dans le contexte d’A-SABR, ces fenêtres de communication sont décrites à l’avance dans un \gls{cp}. Celui-ci indique quels nœuds peuvent communiquer, à quels moments, pendant quelle durée, avec quel débit et quel délai. La bibliothèque A-SABR exploite ensuite ces informations pour calculer des routes adaptées au contexte DTN, en tenant compte des contraintes temporelles et des ressources disponibles. Ainsi, le routage ne consiste pas seulement à trouver un chemin dans un graphe, mais à déterminer une succession de contacts compatibles dans le temps.

\subsubsection{Rust}

\subsubsection{Rust}

Rust est un langage particulièrement adapté à A-SABR, car la bibliothèque manipule de nombreuses structures de données liées au routage : nœuds, contacts, \gls{bundle}, \gls{cp} et \gls{mg}. Ces objets doivent être parcourus, modifiés et parfois partagés entre plusieurs composants, tout en conservant de bonnes performances.

Un premier intérêt de Rust est la gestion explicite des cas d'échec avec les types \texttt{Option} et \texttt{Result}. Dans A-SABR, une fonction peut par exemple échouer parce qu'aucun contact ne permet une transmission, ou parce qu'un \gls{cp} ne peut pas être correctement parsé :

\begin{minted}[fontsize=\small,breaklines,frame=single]{rust}
pub trait Router<NM: NodeManager, CM: ContactManager> {
    fn route(
        &mut self,
        source: NodeID,
        bundle: &Bundle,
        curr_time: Date,
        excluded_nodes: &[NodeID],
    ) -> Result<Option<RoutingOutput<NM, CM>>, ASABRError>;
}
\end{minted}

L'opérateur \texttt{?} permet ensuite de propager simplement une erreur à la fonction appelante :

\begin{minted}[fontsize=\small,breaklines,frame=single]{rust}
let graph = Multigraph::new(contact_plan)?;
\end{minted}

Si \texttt{Multigraph::new} retourne une erreur, celle-ci est automatiquement renvoyée par la fonction courante ; sinon, la valeur construite est récupérée dans \texttt{graph}. Cela permet de garder un code lisible tout en conservant une gestion explicite des erreurs.

Rust est aussi utile grâce à son système de traits et de généricité. Dans A-SABR, plusieurs composants partagent une même interface tout en ayant des comportements différents. C'est le cas des gestionnaires de contacts, qui implémentent le trait \texttt{ContactManager}. Celui-ci définit notamment une méthode de simulation, \texttt{dry\_run\_tx}, et une méthode de planification réelle, \texttt{schedule\_tx}.

\begin{minted}[fontsize=\small,breaklines,frame=single]{rust}
pub trait ContactManager: Debug {
    fn dry_run_tx(
        &self,
        contact_data: &ContactInfo,
        at_time: Date,
        bundle: &Bundle,
    ) -> Option<ContactManagerTxData>;

    fn schedule_tx(
        &mut self,
        contact_data: &ContactInfo,
        at_time: Date,
        bundle: &Bundle,
    ) -> Option<ContactManagerTxData>;

    fn get_original_volume(&self) -> Volume;
    fn try_init(&mut self, contact_data: &ContactInfo) -> bool;
}
\end{minted}

Cette signature est particulièrement intéressante : \texttt{dry\_run\_tx} prend \texttt{\&self}, ce qui correspond à une simulation sans modification de l'état interne du contact, tandis que \texttt{schedule\_tx} prend \texttt{\&mut self}, ce qui indique que la planification modifie réellement l'état du gestionnaire. Les deux méthodes retournent une \texttt{Option<ContactManagerTxData>}, ce qui exprime directement qu'une transmission peut échouer.

Enfin, les macros illustrent la métaprogrammation en Rust : elles permettent de générer du code à la compilation afin d'éviter la duplication. Dans A-SABR, elles sont notamment utilisées pour créer des variantes proches d'un même comportement ou pour décliner une même logique de test sur plusieurs modules.

\begin{minted}[fontsize=\small,breaklines,frame=single]{rust}
macro_rules! create_new_alternative_path_variant {
    ($struct_name:ident, $better_fn:ident) => { ... };
}
\end{minted}

La compilation de la librairie et de l'interface graphique nécessitant divers paquets Linux, notre travail a été 
réalisé soit par WSL, soit directement depuis une distribution Linux.

\subsection{Ecosystème DTN-MTP}
\subsubsection{A-SABR}
A-SABR est la bibliothèque centrale sur laquelle porte notre projet. Elle implémente des mécanismes de routage pour les réseaux \gls{dtn}, en s’appuyant sur les principes du \gls{sabr}. Son rôle est de déterminer comment acheminer un \gls{bundle} d’un nœud source vers une ou plusieurs destinations, non pas à partir d’une connexion permanente, mais à partir des contacts disponibles dans le temps.

Pour cela, A-SABR lit un \gls{cp}, qui décrit les fenêtres de communication prévues entre les nœuds du réseau. Ces informations sont ensuite transformées en \gls{mg}, structure sur laquelle les algorithmes de routage peuvent travailler. La bibliothèque peut alors calculer des routes compatibles avec les contraintes temporelles du réseau, comme les dates de début et de fin des contacts, les délais de propagation, les débits disponibles ou encore la priorité des bundles.

L’intérêt d’A-SABR est aussi sa modularité. Elle permet de comparer différentes stratégies de gestion des contacts, différentes variantes de pathfinding et plusieurs méthodes de routage. Cette architecture rend la bibliothèque adaptée à l’expérimentation : on peut tester plusieurs modèles de transmission, mesurer leurs performances, vérifier leur comportement par des tests, puis les intégrer progressivement dans l’écosystème open source DTN-MTP.

Le projet A-SABR est mené par Monsieur Olivier de Jonckère au sein du LIRMM. Il s’agit d’un projet entièrement open source, ce qui signifie que son développement repose sur un dépôt public, des contributions externes et un processus de validation par Pull Requests. Ce cadre nous a permis de travailler sur un projet réel, maintenu en dehors du contexte strictement universitaire, et d’inscrire nos contributions dans un écosystème logiciel existant.

\subsubsection{Bp-socket}

BP-Socket est un projet visant à fournir une interface de type socket Linux pour le Bundle Protocol, utilisé dans les réseaux \gls{dtn}. L’idée est de permettre à une application d’envoyer et de recevoir des \gls{bundle} avec une API proche de celle utilisée pour des communications classiques en UDP ou TCP.

Dans son fonctionnement initial, BP-Socket repose sur la création d’une nouvelle famille de sockets, \texttt{AF\_BP}, afin de représenter les communications Bundle Protocol au niveau du système. Une application peut alors manipuler une socket BP comme une socket réseau classique, tandis que l’implémentation se charge de transmettre les données sous forme de bundles.

Premièrement, on crée une fonction \texttt{update} tiré de la fonction \texttt{start} de l'interface, celle-ci prend en argument un \texttt{str} et un chemin de fichier.
Il faut aussi modifier la fonction de lancement de l'interface, qui ici se nomme \texttt{try\_init} pour qu'elle prenne également un \texttt{str} et un chemin de fichier, on récupère également les valeurs de 
départ qu'on met dans l'appel de la fonction. \texttt{try\_init} à aussi été modifie pour retourner le nombre de node et de contact trouvée dans le \gls{cp} chargé.\\

La fonction \texttt{update} prend donc en entrée un \texttt{str} qui correspond au nom de l'algorithme utilisé, et 
un Path sous forme de string qui correspondra au chemin jusqu'au \gls{cp}.

Elle va ensuite faire un match sur \texttt{"PredictionConfig::try\_init(path.clone(), algo)"} qui retournera donc un Result,
pour empêcher un crash total de l'interface au moindre soucis.

Si Ok alors on met à jour la configuration qu'on envoie au Frontent et on l'affiche dans l'App Event,
si Err, on retourne l'erreur et on l'affiche dans l'App Event.

Etant basé sur A-SABR, il faut régulièrement faire de légères modifications dûes à des refactorisations de fonctions ou de
l'ajout de Result remplaçant des Options dans la librairie. \\

\subsection[Pipeline]{Pipeline CI/CD}
\subsubsection{Outils de qualité logicielle}
Durant notre travail, l'une de nos tâches a été d'offrir aux différents dépôts des pipelines robustes et utiles aux développeurs de l'équipe.
Pour cela, il a fallu dans un premier temps définir les choses à vérifier grâce au travail de pipelines.
Après discussion avec Monsieur De Jonckère, nous avons décidé de travailler sur le format du code, ses failles de logique ou de sécurité et sur le lancement des tests puis de leur couverture.
Nous présenterons donc dans cette partie, les outils utilisés pour mener à bien cette tâche.

Comme tout projet Rust, nous utilisons ici \texttt{Cargo}. \texttt{Cargo} est l'outil de construction et le gestionnaire de paquets officiel de Rust, 
il permet de gérer les dépendances, compiler le code en offrant un retour intuitif au développeur en cas d'erreur ou encore de standardiser la structure d'un projet Rust.
\texttt{Cargo} possède beaucoup de commandes utiles comme \texttt{cargo build} pour compiler ou \texttt{cargo check} pour vérifier la compilation sans générer de binaire.

Dans le cadre de nos pipelines, nous utilisons d'autres commandes de \texttt{Cargo} dont voici leur explication:
\begin{itemize}
    \item \texttt{cargo fmt --check}: au lancement de cette commande, une vérification de format est lancée pour vérifier si le code respecte les standards de style de la communauté.
    L'ajout de \texttt{--check} impose de faire uniquement une vérification sans modifier le code.
    \item \texttt{cargo clippy -D warnings}: cet outil est le "linter" de Rust, il permet de trouver des erreurs de logique ou des mauvaises pratiques de programmation comme une fonction possédant trop de paramètres.
    En ajoutant \texttt{-D warnings}, on permet de rendre les avertissements bloquants.
    \item \texttt{cargo test}: grâce à cette commande, l'ensemble des tests du projet sont lancés. Les tests unitaires précédés du module \mintinline{rust}|#[cfg(test)]| et les tests d'intégration contenus dans le dossier \texttt{/tests}
\end{itemize}
\subsubsection{Intégration continue (CI)}
Pour ce projet, afin d'utiliser des outils de CI adaptés, nous avons utilisé GitHub Actions, un outil d'automatisation intégré à GitHub qui permet d'effectuer des vérifications sur les travaux poussés vers le dépôt.
Cet outil se base sur 3 choses essentielles:
\begin{itemize}
    \item Le déclencheur: quel événement va déclencher le travail de vérification (\texttt{git push} ou une PR).
    \item Le plan: le fichier crée en \texttt{.yml} permettant d'orchestrer le travail à faire.
    \item Le runner: serveur distant intégré à GitHub qui offre un environnement pour lancer nos vérifications.
\end{itemize}

Afin de rendre notre travail moins permissif à différents types d'erreurs, nous avons couplé les outils de GitHub Actions avec un autre principe, celui des pre-commit.
Nous mettons en place un outil qui nous permet de faire des vérifications avant la création du commit donc en local. Ce qui rend ce principe très utile est sa possibilité d'annuler un commit s'il ne respecte pas 
les règles dictées par le fichier \texttt{.pre-commit-config.yaml}. Si un développeur tente de commit des fichiers contenant des mauvaises pratiques de programmation (erreur clippy),
au moment de taper la commande \texttt{git commit}, l'outil pre-commit annulera la commande.

Pour enrichir notre travail, nous utilisons des \texttt{hooks} qui sont des outils téléchargeables nous permettant de mettre en place de nouvelles vérifications et modifications utiles au projet:
\begin{itemize}
    \item \texttt{trailing-whitespace}: retire les espaces en trop dans le code.
    \item \texttt{end-of-file-fixer}: rajoute un saut de ligne unique vide à la fin de chaque fichier.
    \item  \texttt{check-yaml}: vérifie l'intégrité de nos fichiers de configuration de pipelines.
    \item \texttt{check-added-large-files}: bloque le commit si celui si contient des fichiers trop lourds.
\end{itemize}

Quand tout cela a été mis en place, nous pouvons ajouter nos outils Rust présentés plus tôt. 
Dans la structure décidée avec Monsieur De Jonckère, nous avons fait le choix de faire 3 vérifications avant le commit:
les hooks précedemment cités, le format du code (\texttt{cargo fmt --check}) et les erreurs de logique et pratiques (\texttt{cargo clippy -D warnings}).
Il est important de noter un détail sur ce travail de pre-commit, nous avions 2 choix pour la maintenabilité: utiliser un dépôt en ligne contenant les outils Rust ou

\subsubsection[Module : Contact Manager]{Contact Manager}

Les \textit{Contact Managers} sont les composants responsables de la planification des transmissions sur un contact réseau dans A-SABR. Leur rôle est de déterminer si un bundle peut être transmis à un instant donné, de calculer les dates associées à cette transmission, puis de mettre à jour l’état du contact lorsque celle-ci est effectivement planifiée.

Tous les gestionnaires de contacts implémentent une interface commune reposant principalement sur deux opérations :
\begin{itemize}
    \item \texttt{dry\_run\_tx}, qui simule une transmission sans modifier l’état interne du contact,
    \item \texttt{schedule\_tx}, qui planifie effectivement la transmission et met à jour les ressources disponibles.
\end{itemize}

Plusieurs implémentations de \textit{Contact Manager} coexistent dans le projet afin de proposer différents niveaux de précision et de modélisation. Elles sont regroupées en deux grandes familles : les gestionnaires \textit{Legacy}, reposant sur des modèles historiques plus simples, et les gestionnaires basés sur la segmentation, offrant une représentation plus fine des contacts.

\paragraph{Legacy :}~\\\\
Les gestionnaires \textit{Legacy} correspondent aux premières implémentations de gestionnaires de contacts présentes dans A-SABR. Ils sont considérés comme le standard actuelle. Ils restent présents dans la librairie pour des raisons de compatibilité et de comparaison, mais ne constituent pas l’approche principale utilisée pour représenter finement les transmissions.

Ces gestionnaires restent néanmoins importants à tester, car ils font encore partie du code existant et sont utilisés comme référence pour nos tests.

\begin{figure}[H]
\centering
\includegraphics[width=0.75\textwidth]{Images/legacy_managers_comparison.png}
\caption{Comparaison des comportements des gestionnaires Legacy (ETO, EVL, QD).}- \\
\label{fig:legacy_managers_comparison}
\end{figure}


Le sous-module contient trois implémentations : EVL, QD et ETO. Ces implémentations sont générées à l'aide d'une macro qui prend différents paramètres :

\begin{itemize}
    \item \texttt{\$add\_delay} : indique si le délai de queue doit être pris en compte pour le calcul de \texttt{tx\_start}
    \item \texttt{\$auto\_update} : indique si la queue est mise à jour automatiquement après un \texttt{schedule\_tx} ou manuellement
    \item \texttt{\$prio\_count} : donne le nombre de niveaux de priorités
    \item \texttt{\$with\_budget} : indique si des budgets sont appliqués lorsque des priorités sont présentes
\end{itemize}

EVL, QD et ETO se distinguent par leurs valeurs de ces paramètres. EVL est le manager le plus simple : il se contente de consommer le volume d'un bundle à chaque \texttt{schedule\_tx} sur un contact. La queue est mise à jour automatiquement et il n'y a pas de délai. QD est similaire à EVL, à la différence que \texttt{tx\_start} est décalé depuis le début du contact en fonction du volume déjà réservé. Enfin, ETO est le plus complexe des trois : le décalage de la queue doit être géré manuellement et débute à \texttt{at\_time}, soit le temps courant.

Chacun de ces fichiers instancie la macro pour générer trois variantes : un manager de base sans priorités, un avec priorités, et un autre avec priorités et budgets, un budget étant la quantité maximale de données qu'un bundle peut réserver sur un contact.\\\\



Pour tester ces implémentations, et sachant qu'elles partagent une base de tests en commun, nous avons mis en place des macros de tests dans un fichier \texttt{test\_helpers.rs} afin d'éviter la duplication de code. Quatre macros ont été définies :

\begin{itemize}
    \item \texttt{generate\_common\_tests!} : regroupe les tests communs à tous les managers, en testant le comportement de \texttt{dry\_run\_tx} et \texttt{schedule\_tx} en termes de timing, de volume et de conditions limites.

    \item \texttt{generate\_auto\_update\_tests!} : teste les comportements liés à la mise à jour automatique de la queue, notamment la saturation du contact et l'isolation des priorités. Ces tests ne s'appliquent pas à ETO puisque \texttt{\$auto\_update=false} pour lui.

    \item \texttt{generate\_budget\_tests!} : vérifie que les limites de budget par priorité sont bien respectées.

    \item \texttt{generate\_budget\_auto\_update\_tests!} : teste l'indépendance des budgets entre priorités. Ces tests ont été séparés de \texttt{generate\_budget\_tests!} car ils nécessitent \texttt{auto\_update=true}, ce qui exclut ETO. \\
\end{itemize}

En complément de ces tests communs, des tests spécifiques ont été ajoutés afin de valider les comportements propres à chaque implémentation.

Pour \texttt{ETO}, les tests spécifiques vérifient que la file d’attente n’est pas automatiquement mise à jour après une planification. Ainsi, plusieurs appels successifs à \texttt{schedule\_tx} peuvent produire le même résultat tant qu’aucune mise à jour manuelle de la file n’est effectuée.

Pour \texttt{EVL}, les tests permettent de confirmer que l’occupation de la file d’attente influence la capacité disponible, sans modifier directement la date de début de transmission. Autrement dit, le volume restant diminue, mais \texttt{tx\_start} reste indépendant de cette occupation.

Enfin, pour \texttt{QD}, les tests valident que l’occupation de la file peut décaler la date effective de début de transmission. Ils vérifient également qu’une transmission peut devenir impossible si ce décalage repousse la fin de transmission au-delà de la fenêtre du contact.

L’ajout de ces tests spécifiques permet donc de compléter la couverture existante sur les gestionnaires \textit{Legacy}, tout en documentant plus clairement les différences de comportement entre \texttt{ETO}, \texttt{EVL} et \texttt{QD}.\\

Nous avons aussi corrigé des bugs dans la macro principale qui génère les trois implémentations \textit{Legacy}. 

Avant, lorsqu'un \gls{bundle} avec priorités utilisait les fonctions
\texttt{get\_queue\_size}, \texttt{enqueue} ou encore \texttt{dequeue}
avec un Contact Manager qui ne prend pas en compte les priorités,
le code paniquait.
Désormais, une variante qui ne s'occupe pas des priorités traite
les \glspl{bundle} avec priorités comme s'ils n'en avaient pas.

De plus, dans la variante budgétée de \texttt{generate\_struct\_management!},
le tableau d'initialisation des budgets dans \texttt{build\_parsing\_output}
était hardcodé à 3, ce qui provoquait une erreur de
compilation lorsque la macro était instanciée avec un nombre de priorités
différent de \texttt{3}.
Cela a été corrigé afin qu'il s'adapte correctement au nombre de priorités
avec lequel on l'instancie.

\paragraph{Segmentation :}~\\\\

Le module \textit{Segmentation} propose une modélisation plus fine des contacts que les gestionnaires \textit{Legacy}, en découpant un contact en plusieurs intervalles temporels indépendants. Chaque segment peut ainsi posséder ses propres caractéristiques de transmission, comme le débit ou le délai.
\begin{figure}[H]
\centering
\includegraphics[width=0.5\textwidth]{Images/legacy_vs_segmentation.png}
\caption{Illustration du découpage d’un contact en segments temporels indépendants.}
\label{fig:segmentation_vs_legacy}
\end{figure}

Le gestionnaire de segmentation repose sur trois structures principales : les segments de débit (\texttt{rate\_intervals}), les segments de délai (\texttt{delay\_intervals}) et les intervalles libres (\texttt{free\_intervals}), représentant les portions du contact encore disponibles pour la transmission. Chaque segment est défini par un intervalle temporel et une valeur associée.

Deux opérations principales sont utilisées :
\begin{itemize}
    \item \texttt{dry\_run\_tx} : simulation d’une transmission sans modification de l’état interne,
    \item \texttt{schedule\_tx} : planification effective de la transmission avec mise à jour des intervalles libres.
\end{itemize}

Lorsqu’une transmission est demandée, le gestionnaire recherche un intervalle libre compatible avec l’instant considéré (\texttt{at\_time}). Si cet instant est situé avant le début du contact, la transmission démarre au début du premier intervalle valide ; sinon, elle commence à \texttt{at\_time}. La durée de transmission est ensuite calculée en tenant compte des segments de débit traversés, ce qui permet à une transmission de s’étendre sur plusieurs segments successifs si nécessaire.

\begin{figure}[H]
\centering
\includegraphics[width=0.55\textwidth]{Images/free_intervals_evolution.png}
\caption{Découpage d’un intervalle libre après planification d’un bundle.}
\label{fig:free_intervals}
\end{figure}

La planification d’un bundle entraîne une mise à jour dynamique des intervalles libres. Comme illustré ci-dessus, un intervalle initial peut être découpé en plusieurs sous-intervalles afin de conserver une représentation cohérente des ressources disponibles pour les transmissions futures.

Les tests implémentés dans le fichier \texttt{seg.rs} ont permis de valider plusieurs comportements essentiels :
\begin{itemize}
    \item insertion d’un bundle dans un intervalle libre,
    \item enchaînement de plusieurs transmissions successives,
    \item gestion de débits variables,
    \item prise en compte correcte de l’instant \texttt{at\_time}.
\end{itemize}

Ces tests garantissent la cohérence entre les fonctions \texttt{dry\_run\_tx} et \texttt{schedule\_tx}, ainsi que la mise à jour correcte des intervalles libres après chaque planification.

Les hooks \textit{pre-commit}, incluant \texttt{cargo fmt} et \texttt{cargo clippy}, ont empêché certains commits en raison de problèmes de formatage ou de warnings. Ces difficultés ont été résolues en appliquant les corrections nécessaires et en respectant les règles imposées par le projet.

Ces difficultés ont permis de mieux comprendre les invariants du modèle et de renforcer la robustesse des tests implémentés.\\

    \item[Vérifications: ] la fonction vérifie ces 3 choses:

\subsubsection[Module : Pathfinding]{Pathfinding}
Le rôle de ce module est de calculer, à partir d'un \gls{bundle} et d'un instant donnée, la meilleurs route vers
une destination en tenant compte des contraites temporelles de chaque contact.
Il expose un Trait, \texttt{PathFinding}, implémentée par trois variantes : Node, Contact et Hybride, détaillée ci-après,
ainsi qu'un sous module \texttt{limiting\_contact} qui permet de supprimer temporairement le contact le plus limitant d'une route 
pour forcer la découverte de routes alternatives. 

Chaque variante est instanciée avec une métrique de distance, soit \textbf{Hop} qui minimise le nombre de saut, 
soit \textbf{\gls{sabr}} qui minimise le temps d'arrivée.
\paragraph{Try\_make\_hop :}~\\\\

\texttt{try\_make\_hop} est la fonction centrale du module de pathfinding. Son rôle est de déterminer, depuis un nœud source à un instant donné, le meilleur contact disponible pour acheminer un bundle vers un nœud voisin en un seul saut. Elle prend en entrée un \texttt{RouteStage} source qui représente l'état courant du routage (temps d'arrivée au nœud, délai cumulé, expiration), une liste de contacts triés par ordre de début, ainsi que les nœuds émetteur et récepteur. Elle retourne soit le meilleur saut trouvé, soit \texttt{None} si aucun contact ne permet la transmission.

La fonction parcourt chaque contact fourni en entrée à partir d'un \texttt{first\_contact\_index}, ce qui permet un \textit{lazy pruning} : les contacts déjà dépassés par un saut précédent dans Dijkstra n'ont pas besoin d'être reconsidérés. Pour chaque contact, elle appelle \texttt{dry\_run\_tx} du Contact Manager pour simuler la transmission, puis retient le contact permettant la date d'arrivée \texttt{rx\_end} la plus rapide. Elle s'arrête dès que le début d'un contact est postérieur au meilleur \texttt{rx\_end} déjà trouvé, les contacts étant supposés triés. Si un contact est retenu, elle construit un nouveau \texttt{RouteStage} avec les métadonnées mises à jour : \texttt{hop\_count} incrémenté, \texttt{cumulative\_delay} mis à jour et \texttt{expiration} calculée comme le minimum entre la fin du contact et l'expiration propagée depuis la source.

\texttt{try\_make\_hop} intègre également des \textit{features}, c'est-à-dire des portions de code activées uniquement si la feature est spécifiée au lancement. On trouve ainsi \texttt{contact\_suppression}, \texttt{node\_tx} et \texttt{node\_rx} qui permettent de faire échouer le \texttt{dry\_run} et donc de renvoyer \texttt{None}, ainsi que \texttt{node\_proc} qui permet le traitement du bundle avant l'émission, pouvant décaler le temps d'envoi.

\paragraph{Tests de try\_make\_hop :}~\\\\

Afin de tester cette fonction, nous avons créé un \textit{mock} de \texttt{NodeManager} dans le fichier \texttt{test\_helpers} du dossier pathfinding. En effet, nos tests doivent contrôler le comportement des nœuds émetteurs et récepteurs avec les features \texttt{node\_tx} et \texttt{node\_rx}, contrairement aux autres fichiers qui n'ont besoin que de \texttt{NoManagement}. Ce mock permet d'implémenter et de contrôler un \texttt{NodeManager} avec quatre fonctions : \texttt{accepting()} qui accepte toujours, \texttt{refusing\_rx()} et \texttt{refusing\_tx()} qui rejettent la transmission d'un côté, et \texttt{processing(delay)} qui simule un délai de traitement du bundle. Les fonctions helpers telles que \texttt{make\_node}, \texttt{make\_contact}, \texttt{make\_source} et \texttt{make\_bundle} ont été rendues génériques sur \texttt{NM} pour pouvoir être partagées entre les deux types de manager.

Les tests mis en place couvrent les cas suivants :

\begin{itemize}
    \item \textbf{Cas de base} : appel avec une liste de contacts vide, un \texttt{first\_contact\_index} dépassant le slice, ou un bundle trop volumineux pour un contact donné. On vérifie que la fonction renvoie bien \texttt{None} dans ces cas.

    \item \textbf{Transmission valide} : test avec un seul contact compatible. On vérifie que \texttt{Some} est bien retourné.

    \item \textbf{Lazy pruning} : test avec \texttt{first\_contact\_index=1} qui saute un contact valide. On vérifie que le résultat correspond bien au second contact et non au premier.

    \item \textbf{Feature \texttt{contact\_suppression}} : si tous les contacts sont supprimés, la fonction doit renvoyer \texttt{None}. Si un seul contact parmi deux est supprimé, le second doit bien être choisi.

    \item \textbf{Mock NodeManager (\texttt{node\_tx}, \texttt{node\_rx}, \texttt{node\_proc})} : ces trois tests ne peuvent pas utiliser \texttt{HopContext} car ils nécessitent des managers différents sur \texttt{tx} et \texttt{rx}. On vérifie qu'un nœud refusant l'émission ou la réception bloque le saut, et qu'un délai de traitement décale correctement le \texttt{sending\_time} et donc le \texttt{at\_time} dans le résultat.

    \item \textbf{Sélection du meilleur contact} : test avec 4 contacts dont certains sont élagués par le lazy pruning. On vérifie que le contact avec le \texttt{rx\_end} minimal est bien choisi et que les métadonnées finales sont correctes.

    \item \textbf{Test sur 2 sauts} : le résultat du premier saut est utilisé comme source pour le second. On vérifie que \texttt{expiration} et \texttt{cumulative\_delay} se sont bien propagés et que le meilleur contact est toujours sélectionné avec les bonnes métadonnées.
\end{itemize}
\paragraph{Contact,Hybrid et Node parenting :}~\\\\
Contact, Hybrid et Node sont 3 variantes de l'algorithme de Dijkstra qui sont utilisés pour le routage :\\

Node est le plus simple, il explore le \gls{mg} crée noeud par noeud, et ne maintient qu'une seule route optimale.
Il trouve le chemin optimal selon la métrique choisie mais s'arrête dès que la destination est atteinte en mode Path,
contrairement aux autres variantes qui continuent d'explorer.\\

Contact explore lui le \gls{mg} par contact plutôt que par noeud, il va utiliser une "Work Area", permettant d'explorer les contacts directement sans passer par la logique nœud par nœud, 
ce qui peut trouver des routes avec moins de saut dans certains cas.\\

Hybrid est la plus complète mais également la plus complexe, il va maintenir une liste de potentiel route optimal
tous le long du parcours, puis, grâce au fonction "can\_retain" et "must\_prune", va décider laquelle garder.\\

Chaque algorithme de parenting possède les trois mêmes variantes, générées sous forme de macros. Comme il existe trois approches principales, Node, Contact et Hybrid, cela donne donc neuf variantes possibles au total :

\begin{itemize}
    \item Path : Chemin simple vers une destination, s'arrête dès qu'elle est atteinte.
    \item PathExcl : Si l'on veut exclure un noeud précis.
    \item TreeExcl : Explore l'entièreté du \gls{mg} avec exclusion.\\
\end{itemize}

Les tests sonts simples, on crée plusieurs \gls{mg} auquel on applique chacune des macros,
on utilise le manager le plus simple possible pour ne pas rendre le test trop lourd, on applique en dernier la méthode Hop (le moins de saut) ou la méthode SABR (le plus rapide) 
pour avoir le plus de données de test possibles.

Les résultats attendus sont le nombre de sauts réalisés, et le temps d'arrivée.
Pour les cas d'exclusion, on donne comme objectif par exemple le noeud 3 en excluant le noeud 2 et l'on doit avoir en retour un échec du au noeud exclu.\\

Dans un soucis de lisibilité et de maintenabilité du code, toutes les fonctions de création d'objets (noeud,contact,bundle,multigraph)
sont réunies dans un fichier à part (test\_helpers). De cette manière, seuls les tests sont présents dans leurs fichiers cibles. \\

Prenons un test en particulier et décryptons le : \label{testPathFinding} \ref{CodetestPathFinding} (Cliquer sur le numéro pour être envoyé à l'annexe correspondant) \\

Nous avons donc une première fonction qui va créer le \gls{mg} auquel on va appliquer nos tests, le multigraph crée possède
5 noeuds qui ont un manager neutre (qui ne fait rien) pour faire au plus simple et de 5 contacts.

Exemple : le premier contact émet du noeud 0 au noeud 1, début du contact à 0 et fin à 10, 
débit de 1 unité/seconde et aucun délai de propagation.

Ensuite nous créeons un \gls{bundle} à destination du noeud 4, sans priorité et sans taille précise.

On contruit les deux chemin, un avec la méthode SABR, l'autre avec la méthode Hop, 
puis nous lui appliquons notre fonction de test.\\

\begin{itemize}
    \item Hop : On attend 3 sauts en 50 unité de temps.
    \item SABR : On attend 4 sauts en 50 unité de temps.\\
\end{itemize}

Le fait que le nombre de saut diverge entre Hop et \gls{sabr} permet une plus grand couverture pour détecter une erreur de la librarie.\\

\subsubsection[Module : Routing]{Routing}

Le dossier \texttt{routing} regroupe l'ensemble de la logique de routage du projet. Son rôle est de prendre un bundle en entrée et de retourner un \texttt{RoutingOutput}, c'est-à-dire la liste des premiers sauts à emprunter pour acheminer ce bundle vers sa ou ses destinations.

Le fichier \texttt{mod.rs} de ce dossier définit les bases communes à tous les routeurs. Il contient d'abord le trait \texttt{Router}, qui impose à toute implémentation de routeur de fournir une méthode \texttt{route()} prenant en entrée le nœud source, le bundle à router, le temps courant et une liste de nœuds exclus. Il définit ensuite \texttt{RoutingOutput}, une structure qui associe à chaque premier saut (représenté par un contact) la liste des \texttt{RouteStage} des destinations atteignables via ce saut. Cette structure expose également \texttt{lazy\_get\_for\_unicast()}, une méthode qui recherche le premier saut menant à une destination précise, utilisée notamment dans les tests.

Le fichier contient aussi des fonctions partagées par tous les routeurs, comme \texttt{schedule\_unicast}, \texttt{schedule\_multicast}, \texttt{dry\_run\_unicast\_path}, \texttt{dry\_run\_unicast\_tree} et \texttt{dry\_run\_multicast}. Ces fonctions implémentent deux phases distinctes. Le \textit{dry run} simule le parcours d'un chemin ou d'un arbre sans réserver de ressources, pour vérifier qu'un bundle peut effectivement atteindre sa destination en fonction des contacts. Le \textit{scheduling} effectue ensuite la réservation réelle en appelant \texttt{schedule()} sur chaque \texttt{RouteStage} du chemin.

Le dossier contient trois implémentations concrètes du trait \texttt{Router} : CGR, VolCGR et SPSN. Dans le cadre de notre projet, nous n'avons travaillé que sur VolCGR et SPSN. 

VolCGR est une évolution de CGR qui est la méthode de la NASA, et Spsn qui est une implémentation toute nouvelle venant d'A-SABR.\\

Des premiers tests ont également été développés pour le module routing. 
Cependant, par manque de temps et à la suite d’une PR récente ayant refactorisé la gestion des nœuds et des contacts,
leur intégration propre dans la version finale du projet n’a pas pu être menée à terme. 
Ces travaux ne sont donc pas présentés en détail dans ce rapport, mais restent disponibles sur une branche
dédiée de notre dépôt afin de pouvoir être repris et finalisés ultérieurement.

Il faut donc : 

\begin{itemize}

    \item Nettoyer le daemon et supprimer tous les fichiers du General Netlink.

    \item Définir le nouveau protocole de communication entre le hook et le Daemon, celui ci contiendra toutes les options nécessaires(\texttt{OPEN ENDPOINT, CLOSE ENDPOINT, SEND BUNDLE, RECV BUNDLE}).

    \item Ecrire le hook en lui même.
    
    \item Ecrire un fichier qui gérera l'état des files descripteur crées par notre version de \texttt{socket}

    \item Modifier le daemon avec les nouvelles fonctions relatives au hook (\texttt{send bundle, connect, message})
    
    \item Modifier l'interface \gls{ion} avec les nouvelles fonctions du Daemon.\\\\
\end{itemize}



Il y a beaucoup de code, donc pour rester dans le plus intéressant,
présentons le fichier le plus intéressant, le hook. \label{Result_Hook}\ref{CodeHook}\\

Dans les premières lignes du fichier, on retrouve les signatures des fonctions \texttt{socket,close,bind,sendto}
avec le nom real avant leur nom. Ceci est à mettre en relation avec la fonction \texttt{attribute} juste en dessous.

Cette fonction permet de dire grâce à \texttt{dlsym} : Si j'appelle, par exemple, \texttt{real\_socket},
le dynamic loader ira chercher dans la glibC la fonction s'appelant socket.

De cette manière nous pouvons redéfinir socket, et choisir si oui ou non on appelle la vraie fonction.\\

On suit ensuite le schéma suivant: lorsqu'on appelle une fonction, on vérifie si le domaine du fd est \texttt{AF\_BP},
si c'est le cas, notre hook a le bon comportement, sinon on appelle la vraie fonction de la glibC.\\

\textbf{connect to daemon}: Etablit une connexion avec le Daemon via un Unix Socket.

Crée un socket Unix avec la vraie \texttt{socket()}de la glibc, puis se connecte au fichier \texttt{/tmp/bp\_daemon.sock} .
Retourne le fd de connexion si succès, -1 sinon. Cette fonction est appelée uniquement depuis \texttt{socket()}
quand une socket BP est ouverte.\\

\textbf{socket} :  Intercepte la création d'un socket BP et le redirige vers le daemon au lieu du kernel.

Fonctionnement : Si la famille d'adresse n'est pas \texttt{AF\_BP}, laisse passer normalement à la glibc. 
Sinon, appelle \texttt{connect\_to\_daemon()} pour ouvrir une connexion vers le daemon, 
enregistre ce fd dans \texttt{fd\_table} pour le reconnaître plus tard, et le retourne à l'application.
L'app croit avoir un vrai socket kernel alors qu'elle a en réalité un fd de connexion Unix vers le daemon.\\

\textbf{bind} : Intercepte l'association d'une adresse BP à une socket et demande au daemon d'ouvrir l'endpoint correspondant dans \gls{ion}.

Si le fd n'est pas BP, comportement normal. Sinon, récupère le \textbf{node\_id} et \textbf{service\_id} depuis \texttt{sockaddr\_bp},
les stocke dans \texttt{fd\_table}, puis envoie un message \textbf{BP\_IPC\_OPEN\_ENDPOINT} au daemon.
Attend la réponse OK ou ERR. Si OK, \gls{ion} a ouvert l'endpoint et est prêt à envoyer/recevoir sur \textbf{ipn:node\_id.service\_id}.\\

\textbf{close} : Intercepte la fermeture d'un socket BP et demande au daemon de fermer l'endpoint dans \gls{ion}.

Si le fd n'est pas BP, comportement normal. Sinon, envoie \textbf{BP\_IPC\_CLOSE\_ENDPOINT} au daemon avec le \textbf{node\_id} et \textbf{service\_id} stockés dans \texttt{fd\_table}. 
Attend la confirmation, puis supprime l'entrée de \texttt{fd\_table} et ferme le fd Unix socket vers le daemon. 
Sans ce \texttt{close()}, ION garderait l'endpoint ouvert indéfiniment et bloquerait les prochains \texttt{bind()},
ce qui nous ait arrivée lors du développement.\\

\textbf{sendto} :  Intercepte l'envoi d'un bundle BP et le transmet au daemon qui le passera à ION.

Si le fd n'est pas BP, comportement normal. Sinon, construit un message \textbf{BP\_IPC\_SEND\_BUNDLE} contenant les adresses
source et destination (\textbf{src\_node\_id, src\_service\_id, dst\_node\_id, dst\_service\_id}),
les flags BP (\textbf{MSG\_ACK\_REQUESTED etc.}) et la taille du payload. 
Envoie d'abord ce header au daemon, puis le payload (le contenu du message) juste après. 
Attend la réponse OK ou ERR et retourne le nombre de bytes envoyés si succès.\\

Le sender est parfaitement fonctionnel, il permet d'envoyer des messsages d'\gls{ion} vers µDTN et ceux sans module kernel.



Il reste le fait de recevoir un message, qui inclut donc le fait de détruire un bundle une fois reçu et lu, mais
par manque de temps nous ne pourrons pas le faire. Néanmoins, Monsieur De Jonckère est en lien avec collègue qui aimerait reprendre le travail
donc nous lui transmettrons nos avancées dès que possible.

\section{Algorithmes et Structures de Données}
\subsection{PrioritySegmentation}
Dans cette partie, nous allons détailler le fonctionnement algorithmique du module \texttt{PrioritySegmentation (pseg)}.
Pour commencer, étudions la fonction de simulation du module \texttt{dry\_run\_tx}. 
Parlons d'abord de sa signature, la fonction prend 4 paramètres:
\begin{itemize}
    \item \&self: la référence vers l'instance du PSegmentationManager contenant toute la logique. Un détail a son importance ici,
    comme la fonction ne fait qu'une simulation on limite l'attribut à une lecture seule en ne mettant pas le mot clé \texttt{mut} indiquant à Rust que cette variable n'est pas mutable.
    \item contact\_data: \&ContactInfo: c'est une \texttt{struct} contenant des attributs \texttt{source} et \texttt{dest} de type \texttt{NodeID}
    et deux attributs \texttt{start} et \texttt{end} de type \texttt{Date}. Cette structure de données représente les limitations physiques: \texttt{source} et \texttt{dest} sont les noeuds concernés par le contact
    et \texttt{start} et \texttt{end} représentent la fenêtre temporelle du contact.
    \item at\_time: Date: c'est l'instant t à partir duquel nous souhaitons insérer le paquet.
    \item bundle: \&Bundle: la référence vers le paquet que l'on souhaite transmettre.
\end{itemize}
Le type de données renvoyé par la fonction est \texttt{Option<ContactManagerTxData>}. En Rust, \texttt{Option<T>} est une énumération polymorphique qui se définit par deux états, None et Some(T). Dans notre cas voici comment sont traités les deux cas: 
\begin{itemize}
    \item Le cas d'échec symbolisé par \texttt{None}: la transmission est impossible.
    \item Le cas de de succès symbolisé ici par \texttt{Some(ContactManagerTxData)}: un créneau utilisable a été trouvé par la fonction. On renvoie donc un objet de type \texttt{struct ContactManagerTxData}
    qui comprend toutes les informations du paquet inséré: les heures de début et de fin d'émission représentés par \texttt{tx\_start} et \texttt{tx\_end}, les heures de réception prenant en compte le délai de propagation représentés par \texttt{rx\_start} et \texttt{rx\_end}
    et la date d'expiration du paquet représenté par le champ \texttt{expiration}.
\end{itemize}

Passons maintenant au corps de l'algorithme. On commence par initialiser deux variables: \texttt{tx\_start} initialisée à \texttt{at\_time}
et \texttt{tx\_end\_opt} représentant l'état de la recherche, \texttt{None} si on cherche un point de départ et \texttt{Some(date)} sinon.
Ensuite, grâce à une boucle \texttt{for}, l'algorithme évalue chaque segment de \texttt{self.booking} avec une complexité de \texttt{O(n), n} étant le nombre de segments.
Dans un premier temps, l'algorithme applique 2 vérifications, avec la ligne \mintinline{rust}{if seg.end <= at_time { continue; }} on ignore les segments qui se trouvent avant du paramètre \texttt{at\_time}.
Grâce à la ligne \mintinline{rust}{if bundle.priority <= seg.val { tx_end_opt = None; continue; }}, on ignore également les segments dont la priorité est supérieur à celle du paquet à insérer.

Après ça, on peut commencer le calcul débutant par \texttt{match tx\_end\_opt}:
\begin{itemize}
    \item Cas \texttt{None}: un espace libre a été trouvé, on fixe \texttt{tx\_start} à \texttt{at\_time} et on calcule les besoins du paquet.
    Si tout rentre dans le segment en cours, on peut renvoyer le résultat, sinon on passe dans l'état \texttt{Some(tx\_end)} pour aller chercher de l'espace dans le ou les segments suivants.
    \item Cas \texttt{Some(tx\_end)}: on cherchait déjà à insérer dans le segment précédent donc on cherche de l'espace supplémentaire dans ce segment. Si \texttt{tx\_end} tombe dans ce segment, on renvoie le résultat sinon on répète l'opération.
\end{itemize}

Si aucun créneau n'a été trouvé, on renvoie None sinon on renvoie la \texttt{struct ContactManagerTxData} correspondante. 
Dans le pire des cas, on a une complexité de l'ordre de \texttt{O(n * r)} avec n le nombres de segments \texttt{booking} er r le nombre de segments de débit.
En effet, à chaque fois qu'un créneau potentiellement libre est trouvé, le calcul de la fin de transmission est relancé par \mintinline{rust}|super::get_tx_end|, cette fonction doit reparcourir les r intervalles de débit.
Donc dans le pire cas, cette sous-recherche est lancée à chaque nouveau segment libre soit une complexité \texttt{O(n * r)}.

Cependant, il est intéressant de noter que physiquement le délai est une valeur qui reste fixe dans la plupart des cas, il sera donc représenté dans la majorité des cas par un segment unique.
Donc la plupart du temps, on aura plutôt une compelexité de l'ordre de \texttt{O(n)}.
\\[1cm]
Après la fonction de simulation, nous pouvons passer à la fonction d'allocation \texttt{schedule\_tx}.
La signature de la fonction est très similaire à celle \texttt{dry\_run\_tx} mais à une différence près: on utilise \texttt{\&mut self} à la place de \texttt{\&self}.
Là où la première fonction n'était qu'une simulation et n'avait pas besoin de modifier des données, ici il faut expliciter le fait que cette variable est mutable et sera donc
modifié par \texttt{schedule\_tx}. Le type de données renvoyé reste également le même.
Passons directement à l'étude algorithmique. Pour commencer, on récupère le résultat de la simulation via \mintinline{rust}|let out = self.dry_run_tx(contact_data, at_time, bundle)?;|. 
L'opérateur '?' est quelque chose de propre à Rust, il permet d'arrêter l'exécution si None est renvoyé.
Après ça, on peut commencer le découpage des segments. Grâce à une boucle \texttt{while} on parcourt tous les segments afin de trouver tous ceux qui sont dans la fenêtre temporelle \texttt{tx\_start -- tx\_end}, avec une complexité \texttt{O(n)}.
Pour chaque segment, il y a 3 opérations possibles:
\begin{itemize}
    \item La coupe à gauche avec \mintinline{rust}{seg.start < tx_start}: si le segment commence avant le début de la fenêtre, il est scindé en deux parties.
    On insère un "nouveau" segment gauche intact grâce à \mintinline{rust}{insert(i)} et on repousse le début du segment actuel à \texttt{tx\_start}.
    \item La coupe à droite avec \mintinline{rust}{seg.end > tx_end}: le principe est très similaire au cas précédent mais avec le côté droit. Si le segment s'arrête après la fin de notre fenêtre,
    on crée un "nouveau" segment droit et on raccourci la fin du segment à \texttt{tx\_end}.
    \item L'écrasement: maintenant que l'on possède un segment qui rentre dans notre fenêtre, on peut simplement remplacer sa valeur de priorité par celle du paquet à insérer grâce à \mintinline{rust}{self.booking[i].val = bundle.priority}
\end{itemize}

Enfin, une procédure simple parcourt tous les segments nouvellement crées sauf le dernier grâce à une boucle \texttt{while j < self.booking.len()-1 (O(n-1))} 
afin d'agréger les segments adjacents de même priorité. Pour chaque segment i, on vérifie si le segment i+1 commence quand le segment i se termine (\mintinline{rust} {if self.booking[j].end == self.booking[j + 1].start}) et s'ils ont la même priorité.
Si c'est le cas on crée un segment avec la date de début du segment i, la date de fin du segment i+1 et leur priorité commune. Si la fusion est faire on n'augmente pas l'indice j, en effet le segment fusionné nouvellement crée à la possibilité d'être fusionné avec le suivant.
S'il n'y a pas de fusion, on passe au segment suivant en faisant j+=1.

Concernant la complexité, le temps de recherche est de l'ordre de \texttt{O(n)}. Les modifications faites pendant cette recherche comme \mintinline{rust}|Vec::insert()| et \mintinline{rust}|Vec::remove()| ont aussi un coût \texttt{O(n)}.
On a donc dans le pire des cas, une complexité \texttt{O(n²)} pour \texttt{schedule\_tx}.
\subsection{HybridParenting}

Dans cette partie, nous allons détailler le fonctionnement algorithmique du module \texttt{HybridParenting},
qui constitue une variante avancée de l'algorithme de Dijkstra implémentant le \textit{Multi-Path Tracking} (MPT).

\subsubsection*{Structures de données}

Avant d'aborder les algorithmes, il est important de comprendre les structures utilisées.
Le module repose sur le trait \texttt{HybridParentingOrd<NM, CM>}, qui impose deux méthodes à implémenter~:
\begin{itemize}
    \item \texttt{can\_retain(prop, known) -> bool}~: détermine si une proposition de route peut être conservée
    même si elle est dominée sur le critère principal. Par exemple dans A-SABR, une route avec
    un nombre de sauts plus faible peut être retenue même si son temps d'arrivée est plus tardif.
    \item \texttt{must\_prune(prop, known) -> bool}~: détermine si une route connue doit être supprimée
    suite à l'insertion d'une proposition.
\end{itemize}


La structure centrale est \texttt{HybridParentingWorkArea<NM, CM>}, qui sert de zone de travail pendant
le calcul des routes. Contrairement au \texttt{PathFindingOutput} classique qui ne stocke qu'une seule
route par destination via un \mintinline{rust}{Vec<Option<SharedRouteStage>>},
la zone de travail maintient un \mintinline{rust}{Vec<Vec<SharedRouteStage>>}~: un vecteur de routes
ordonné par préférence pour chaque destination. Cela permet de suivre plusieurs chemins candidats
simultanément par nœud, ce qui est le principe du MPT. La méthode
\texttt{into\_pathfinding\_output()} convertit ensuite cette structure en \texttt{PathFindingOutput}
standard en ne conservant que le meilleur candidat (index 0) pour chaque destination.

\subsubsection*{Algorithme \texttt{try\_insert}}

La fonction \texttt{try\_insert} est le cœur du mécanisme MPT. Elle tente d'insérer une proposition
de route dans la liste ordonnée des routes connues pour un nœud destination donné.
Sa signature prend deux paramètres~:
\begin{itemize}
    \item \texttt{proposition: RouteStage<NM, CM>}~: la nouvelle proposition de route à évaluer.
    \item \texttt{tree: \&mut HybridParentingWorkArea<NM, CM>}~: la zone de travail contenant
    les listes de routes par destination.
\end{itemize}
Elle retourne un \texttt{Result<Option<SharedRouteStage<NM, CM>>, ASABRError>}.
Le \mintinline{rust}{Option} vaut \mintinline{rust}{Some(route)} si la proposition a été insérée,
\mintinline{rust}{None} sinon. En Rust, \texttt{Result<T, E>} est une énumération qui modélise
la réussite (\mintinline{rust}{Ok(T)}) ou l'échec (\mintinline{rust}{Err(E)}), ici utilisée pour
propager les erreurs d'emprunt (\texttt{ASABRError}).

Passons au corps de l'algorithme. On commence par récupérer la liste des routes connues pour le nœud
destination de la proposition~: \mintinline{rust}{let routes_for_rx_node = &mut tree.by_destination[proposition.to_node as usize]}.
On initialise également deux variables~: \texttt{insert\_index} à la taille de la liste (cas d'insertion
en fin de liste) et \texttt{insert} à \mintinline{rust}{false}.

Si la liste est vide, on insère directement la proposition et on retourne. Sinon, on parcourt les routes
existantes avec une boucle \texttt{for} selon trois cas via \mintinline{rust}{match D::cmp(&proposition, &route_borrowed)}~:
\begin{itemize}
    \item \mintinline{rust}{Ordering::Less}~: la proposition est strictement meilleure sur le critère
    principal (par exemple, temps d'arrivée plus court). On fixe \texttt{insert\_index = idx} et
    \texttt{insert = true} et on sort de la boucle.
    \item \mintinline{rust}{Ordering::Equal}~: doublon détecté, on pose \texttt{insert = false}
    et on sort.
    \item \mintinline{rust}{Ordering::Greater}~: la proposition est moins bonne sur le critère principal.
    On appelle alors \mintinline{rust}{D::can_retain(&proposition, &route_borrowed)}~: si vrai,
    la proposition peut malgré tout être utile (critère lexicographique secondaire) et on continue
    l'itération~; sinon, \texttt{insert = false} et on sort.
\end{itemize}

Si \texttt{insert} est vrai, on procède en trois étapes~:
\begin{enumerate}
    \item \textbf{Détection des routes à éliminer}~: on cherche le premier indice \texttt{truncate\_index}
    à partir de \texttt{insert\_index} où \mintinline{rust}{D::must_prune(&proposition, route)} est vrai,
    ce qui délimite les routes que la nouvelle proposition rend obsolètes.
    \item \textbf{Désactivation}~: les routes à partir de \texttt{truncate\_index} sont marquées
    \mintinline{rust}{is_disabled = true} via leur référence partagée \texttt{Rc<RefCell<...>>},
    ce qui permet de les ignorer si elles se trouvent encore dans la file de priorité.
    \item \textbf{Troncature et insertion}~: on tronque la liste avec \mintinline{rust}{truncate()}
    puis on insère la proposition à \texttt{insert\_index} avec \mintinline{rust}{Vec::insert()}.
\end{enumerate}

Concernant la complexité de \texttt{try\_insert}~: le parcours de la liste est en \texttt{O(k)}
avec $k$ le nombre de routes stockées pour la destination. Les opérations \mintinline{rust}{Vec::insert()}
et \mintinline{rust}{Vec::truncate()} ont également un coût en \texttt{O(k)}. La complexité totale
de \texttt{try\_insert} est donc en \texttt{O(k)}.

\subsubsection*{Algorithme \texttt{get\_next}}

La méthode \texttt{get\_next} implémente la boucle principale de Dijkstra MPT. Sa signature
est identique à celle du trait \texttt{Pathfinding<NM, CM>} et prend quatre paramètres~:
\begin{itemize}
    \item \texttt{\&mut self}~: référence mutable vers l'instance, nécessaire pour emprunter le graphe.
    \item \texttt{current\_time: Date}~: le temps courant à partir duquel démarrer le calcul.
    \item \texttt{source: NodeID}~: l'identifiant du nœud source.
    \item \texttt{bundle: \&Bundle}~: le paquet à router.
    \item \texttt{excluded\_nodes\_sorted: \&[NodeID]}~: liste triée des nœuds à exclure.
\end{itemize}

L'algorithme débute par l'initialisation~: on crée un \texttt{HybridParentingWorkArea} et une
\mintinline{rust}{BinaryHeap<Reverse<DistanceWrapper>>} servant de file de priorité min-heap.
Le nœud source est ajouté à la fois dans \texttt{tree.by\_destination} et dans la file.

La boucle principale \mintinline{rust}{while let Some(Reverse(DistanceWrapper(from_route, _))) = priority_queue.pop()}
extrait la meilleure route courante. Si celle-ci est marquée \mintinline{rust}{is_disabled}, on la
saute~: c'est le mécanisme de \textit{lazy deletion} évitant de retirer explicitement des éléments
de la file. Si le mode \textit{path} (non-tree) est activé et que le nœud extrait est la destination,
on s'arrête immédiatement.

Pour chaque nœud \texttt{receiver} accessible depuis \texttt{from\_route}, on enchaîne trois
appels en \textit{let-chain}~:
\begin{enumerate}
    \item \mintinline{rust}{receiver.lazy_prune_and_get_first_idx(current_time)}~: obtient le premier
    contact non expiré vers ce récepteur.
    \item \mintinline{rust}{try_make_hop(...)}~: construit la proposition de route en calculant le
    temps d'arrivée via le contact sélectionné.
    \item \mintinline{rust}{try_insert::<NM, CM, D>(route_proposition, &mut tree)}~: tente d'insérer
    la proposition dans la zone de travail.
\end{enumerate}
Si les trois réussissent, la nouvelle route est ajoutée à la file de priorité.

En fin de boucle, on tronque chaque liste de \texttt{by\_destination} à un seul élément~:
\mintinline{rust}{v.truncate(1)}, puis on convertit la zone de travail en \texttt{PathFindingOutput}
via \texttt{into\_pathfinding\_output()}.

Concernant la complexité de \texttt{get\_next}~: le Dijkstra classique sur $V$ nœuds et $E$ arêtes
avec une min-heap a une complexité de $O((V + E) \cdot \log E)$. Ici, chaque relaxation
d'arête appelle \texttt{try\_insert} en $O(k)$, ce qui donne une complexité globale de
$O((V \cdot k + E \cdot k) \cdot \log E)$ soit $O(k \cdot (V + E) \cdot \log E)$.
En pratique, $k$ reste petit (le nombre de routes lexicographiquement non dominées par destination
est généralement faible), ce qui rend l'algorithme très proche du Dijkstra standard.

Le module expose trois variantes générées par la macro \mintinline{rust}{define_mpt!}~:
\texttt{HybridParentingTreeExcl} (arbre complet avec exclusions), \texttt{HybridParentingPath}
(chemin simple sans exclusions) et \texttt{HybridParentingPathExcl} (chemin simple avec exclusions).

\section{Analyse des résultats}
\subsection{Benchmark}
Benchmarker une librairie de routage DTN comme A-SABR présente plusieurs défis inhérents.
Contrairement à des algorithmes classiques facilement isolables,
le routage DTN implique une chaîne complète de composants interdépendants
— le parsing du Contact Plan, la construction du Multigraph, le pathfinding, et la gestion des routes — 
ce qui rend difficile l'isolation d'un composant précis pour en mesurer les performances.

Des tentatives de benchmarks réalisés avec Criterion souffrent également du bruit inhérent aux runners GitHub Actions partagés,
où les variations entre deux runs peuvent atteindre ±15\% sans aucun changement de code.
Cela rend la détection de régressions fines impossible en CI.

Une autre tentative, toujours avec critérion mais cette fois ci en local, qui elle génères des schémas
violin plots, bien que lisibles, manquent de conclusivité. Les distributions sont larges et se chevauchent 
partiellement, rendant difficile l'affirmation catégorique qu'une variante est strictement meilleure qu'une autre.
Par exemple la comparaison entre Node, Hybrid et Contact Parenting montre des tendances cohérentes avec la théorie,
mais les intervalles de confiance se superposent trop pour conclure avec certitude.

Enfin le choix d'un seul Contact Plan de référence (\texttt{"A-SABR/benches/ptvg\_files/sample1.json"}) 
avec une unique paire source/destination limite la portée des résultats — un seul scénario peut être 
favorable ou défavorable à une variante particulière sans que ce soit représentatif du comportement général 
de la librairie.\\

\subsection{Organisation réelle du travail}
Après avoir débuté notre travail, nous avons rapidement compris que les choix
d'organisation que nous avons fait allaient devoir rapidement être adaptés afin 
d'être plus efficace. 
\\[1pt]

Pour commencer, nous voulions suivre une structure Gitflow, donc travailler dans des branches feature/
puis mettre en commun dans la branche develop pour ensuite envoyer une version stable vers notre main puis
vers le dépôt de Monsieur De Jonckère. Mais en commençant notamment nos tests, nous nous sommes vite rendus
compte que la mise en commun dans develop n'était pas nécessaire puisque nos tâches ne dépendaient pas de la réussite des autres
Cette approche était même plus compliquée à manipuler pour notre encadrant: à chaque Pull Request de notre main
vers le sien, il aurait dû vérifier un très grand nombre de modifications en une seule fois.
\\[0.5pt]

La solution trouvée a donc été la suivante: notre main suivra toujours la branche main du dépôt d'origine,
à chaque nouvel ajout nous créeons une branche depuis le main, sur cette branche on effectue un \texttt{git rebase main}
pour récupérer le travail le plus récent et nous commençons à travailler. Quand le travail est jugé satisfaisant,
nous ouvrons une Pull Request vers le main du dépôt d'origine afin d'avoir des retours. Après la prise en compte des suggestions,
la Pull Request est acceptée et notre travail est merge dans le dépôt d'origine.

        match PredictionConfig::try_init(path.clone(), algo){
            Ok(update_config) => {
                self.a_sabr = ASabrInitState::Enabled(update_config);

pub fn try_init(cp_path: String, algo : &str) -> io::Result<Self> {
        let cp = IONContactPlan::parse::<NoManagement, EVLManager>(&cp_path)?;

        let nodes_length = cp.nodes.len();
        let contacts_length = cp.contacts.len();

        let node_index_map: HashMap<String, NodeID> = cp.nodes
            .iter()
            .enumerate()
            .map(|(index, node)| (node.get_node_name().to_string(), index as NodeID))
            .collect();

        let router_box = build_generic_router::<NoManagement, EVLManager>(
            algo,
            cp,
            None,
        ).expect("Failed to build router");

        let router: Box<dyn Router<NoManagement, EVLManager> + Send + Sync> =
            unsafe { std::mem::transmute(router_box) };
        // in seconds
        let cp_start_time = DTChatTime::now().timestamp_millis() as f64 / 1000.0;

        Ok(PredictionConfig {
            ion_to_node_id: node_index_map,
            router,
            cp_start_time,
            nodes_length,
            contacts_length,
        })
    }

#define _GNU_SOURCE
#include "bp_ipc.h"
#include "bp_socket.h"
#include "fd_table.h"
#include <dlfcn.h>
#include <sys/socket.h>
#include <sys/un.h>
#include <unistd.h>

static int (*real_socket)(int, int, int) = NULL;
static int (*real_close)(int) = NULL;
static int (*real_bind)(int, const struct sockaddr*, socklen_t) = NULL;
static ssize_t (*real_sendto)(int, const void*, size_t, int, const struct sockaddr*, socklen_t) = NULL;

static void __attribute__((constructor)) init()
{
	real_socket = dlsym(RTLD_NEXT, "socket");
	real_close = dlsym(RTLD_NEXT, "close");
	real_bind = dlsym(RTLD_NEXT, "bind");
	real_sendto = dlsym(RTLD_NEXT, "sendto");
}

static int connect_to_daemon()
{
	struct sockaddr_un addr;
	int fd = real_socket(AF_UNIX, SOCK_STREAM, 0);
	if (fd < 0)
		return -1;

	addr.sun_family = AF_UNIX;
	strncpy(addr.sun_path, BP_IPC_SOCKET_PATH, sizeof(addr.sun_path));

	if (connect(fd, (struct sockaddr*)&addr, sizeof(addr)) < 0) {
		real_close(fd);
		return -1;
	}
	return fd;
}

int socket(int domain, int type, int protocol)
{
    if (domain != AF_BP)
        return real_socket(domain, type, protocol);

    int unix_fd = connect_to_daemon();
    if (unix_fd < 0) {
        return -1;
    }

    fd_table_add(unix_fd, unix_fd, 0, 0);
    return unix_fd;
}

int bind(int sockfd, const struct sockaddr *addr, socklen_t addrlen)
{
    if (!is_bp_fd(sockfd))
        return real_bind(sockfd, addr, addrlen);

    struct sockaddr_bp *bp_addr = (struct sockaddr_bp *)addr;
    bp_fd_entry_t *entry        = fd_table_get(sockfd);

    entry->node_id    = bp_addr->bp_addr.ipn.node_id;
    entry->service_id = bp_addr->bp_addr.ipn.service_id;

    bp_ipc_msg_t msg = {
        .cmd            = BP_IPC_OPEN_ENDPOINT,
        .src_node_id    = entry->node_id,
        .src_service_id = entry->service_id,
        .payload_len    = 0,
    };
    write(entry->unix_fd, &msg, sizeof(msg));

    bp_ipc_response_t resp;
    read(entry->unix_fd, &resp, sizeof(resp));

    if (resp.status == BP_IPC_OK)
        return 0;

    return -1;
}

int close(int fd)
{
    if (!is_bp_fd(fd))
        return real_close(fd);

    bp_fd_entry_t *entry = fd_table_get(fd);

    bp_ipc_msg_t msg = {
        .cmd            = BP_IPC_CLOSE_ENDPOINT,
        .src_node_id    = entry->node_id,
        .src_service_id = entry->service_id,
        .payload_len    = 0,
    };
    write(entry->unix_fd, &msg, sizeof(msg));

    bp_ipc_response_t resp;
    read(entry->unix_fd, &resp, sizeof(resp));

    fd_table_remove(fd);
    real_close(fd);

    return 0;
}

ssize_t sendto(int sockfd, const void* buf, size_t len, int flags,
    const struct sockaddr* dest_addr, socklen_t addrlen)
{
	if (!is_bp_fd(sockfd))
		return real_sendto(sockfd, buf, len, flags, dest_addr, addrlen);

	struct sockaddr_bp* bp_dest = (struct sockaddr_bp*)dest_addr;
	bp_fd_entry_t* entry = fd_table_get(sockfd);

	bp_ipc_msg_t msg = {
		.cmd = BP_IPC_SEND_BUNDLE,
		.src_node_id = entry->node_id,
		.src_service_id = entry->service_id,
		.dst_node_id = bp_dest->bp_addr.ipn.node_id,
		.dst_service_id = bp_dest->bp_addr.ipn.service_id,
		.flags = (uint32_t)flags,
		.payload_len = (uint32_t)len,
	};
	write(entry->unix_fd, &msg, sizeof(msg));
	write(entry->unix_fd, buf, len);

	bp_ipc_response_t resp;
	read(entry->unix_fd, &resp, sizeof(resp));

	if (resp.status == BP_IPC_OK)
		return (ssize_t)len;

	return -1;
}
